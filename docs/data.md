# Data

SEAGym separates task identity, split membership, and run-time batch planning.

```text
TaskIndex -> SplitManifest -> BatchPlan -> ExecutionEngine
```

## Task Index

A task index is a stable list of task references and task attributes. For
Harbor-compatible tasks, SEAGym stores references to Harbor tasks rather than
copying task instructions, environments, solutions, or verifiers.

Minimal record:

```json
{
  "task_id": "task_0001",
  "source": {
    "type": "harbor",
    "dataset": "terminal-bench",
    "task_name": "example-task"
  },
  "attributes": {
    "domain": "code",
    "source_benchmark": "terminal-bench"
  },
  "scoring": {
    "reward_key": "reward",
    "success_threshold": 1.0
  }
}
```

Task indexes may also be generated by scanning a local Harbor-compatible
dataset directory.

## Split Manifest

Split manifests only define conventional base pools:

```json
{
  "split_id": "example_split",
  "seed": 42,
  "splits": {
    "train": ["task_0001", "task_0002"],
    "val": ["task_0003"],
    "test": ["task_0004"]
  }
}
```

They do not define batch size, validation frequency, replay policy, OOD views,
models, budgets, or Harbor job config.

## BatchPlan

`SEAGymDataModule` materializes a `BatchPlan` from the task index, split
manifest, and run config. It contains:

- `train_batches`: ordered lists of train task ids.
- `views.update_validation`: frozen validation tasks.
- `views.final.*`: final evaluation views.
- optional replay, OOD, transfer, or negative-transfer views.

When `dataloader.batch_plan_path` points to an existing file, SEAGym reuses the
stored task ids while preserving the current run identity. This supports paired
comparisons without sharing output directories.

## Artifact Boundary

SEAGym records task ids and source references. Harbor remains responsible for
task content, execution environments, verifiers, parallel trials, and raw task
artifacts.
